145// These are in one struct because the allocator and the memory used in the
146// base map are linked. You neeed to keep the allocators and the map together
147// because you'll need to manipulate the memory allocated with the same
148// allocator. NOTE: I tried to make this structure flat, using arrays to hold
149// differently typed values, but I ran into a problem. I couldn't know how deep
150// an item was in a structure. To search for a named value, I could look
151// through the array of all the names, but that array didn't have the nesting
152// level of the name. I could add a nesting level, but that could make search
153// times bad for an unsuccessful linear search. I may go back to having a root
154// map, and storing items in tagged unions. I'll probably have to use void* to
155// nest maps in maps. Making maps self referential is difficult otherwise. I
156// had an epiphany. Only the names need to be nested. The values don't. I can
157// store all the blobs and types and values in one array, with indicies
158// referencing it if necessary. I'm not sure how efficient that will be though.
159// I need to think about this further. There's only an extra storage cost if I
160// add an extra pointer to something. BLOBS don't cost extra because I need to
161// store data behind a pointer anyway. A reference to a map might cost extra
162// because that could be an index into an array of maps. Same for arrays. There
163// will be pointers to those, but they could be indexes into an array of
164// arrays.
